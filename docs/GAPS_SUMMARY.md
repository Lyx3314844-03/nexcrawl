# OmniCrawl 缺陷总结

## 一句话结论

当前 OmniCrawl 的主要问题已经从“缺少基础设施”转向“边界表达不清 + 大文件维护成本高”。

## 当前仍然成立的问题

### P0

- 文档仍有历史漂移，尤其是 `docs/` 内部对 observability 和能力成熟度的描述
- `server.js` / `job-runner.js` / `reverse-lab-manager.js` 仍然过大

### P1

- coverage 有脚本但缺少稳定基线
- 全局配置已部分接线，但仍非全覆盖
- 高级反爬/逆向能力缺少成熟度标注

### P2

- 部署与运维文档仍不够系统
- 插件/扩展生态文档仍可继续补强

## 已经不应再写成缺陷的事项

以下这类“基础设施完全缺失”的旧口径已经过时：

- 将 metrics / observability 基础面描述为不存在
- 将安全工作流描述为不存在
- 将贡献指南描述为不存在
- 将结构化错误边界描述为不存在
- 将日志脱敏描述为不存在
- 将根测试脚本描述为不存在

这些都已在当前仓库中落地，文档应以“边界/成熟度”而不是“完全缺失”来描述。

## 推荐下一步

1. 继续清理 `docs/` 的剩余过时表述
2. 拆 `server.js`
3. 拆 `job-runner.js`
4. 建立 coverage 基线
